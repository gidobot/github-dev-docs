OA Mapper — Technical Review

Occupancy-Map Obstacle Avoidance · ACFR Seeker AUV · acfr-lcm

Overview

The OA Mapper is a probabilistic obstacle-avoidance system added to acfr-lcm for the ACFR Seeker AUV. It maintains a 2-D voxel occupancy grid anchored to the vehicle in the along-heading × depth plane, fuses observations from three onboard sensors, and produces a continuous vertical depth command that keeps the vehicle at a constant altitude above terrain while safely crossing cliffs and avoiding obstacles ahead.

Core components

  • Core library: auv_oavoid::OccupancyMap and ObstacleMapper in src/acfr/oa-mapper/occupancy_map.* — LCM-free C++ library containing all grid logic, sensor updates, and planning. No framework dependency.
  • LCM node: oa_mapper.cpp wraps the core library: subscribes to sensor and nav LCM channels, calls the planning pipeline on every update, and publishes OA_COMMAND + OA_GRIDMAP at the configured rate.
  • Web visualizer: src/acfr/python/oa-map-visualizer.py subscribes to OA_GRIDMAP and serves a live 3-panel view (top-down grid, depth profile, 3-D terrain) at http://localhost:8050.

Architecture

The OA mapper runs as a standalone process alongside the existing vehicle software stack. It is purely reactive — consuming sensor and nav LCM messages and publishing commands — with no shared memory or direct coupling to the planner.

System components and data flow

Onboard sensors (publish to LCM bus):

  • DVL → <Vehicle Name>.NUCLEUS.BOTTOMTRACK
  • Altimeter → <Vehicle Name>.NUCLEUS.ALTIMETER
  • Forward sonar → <Vehicle Name>.ISA500_FWD
  • Navigation → <Vehicle Name>.ACFR_NAV
  • Heartbeat → HEARTBEAT_5HZ (dashed — rate gate only)

oa-mapper (central process): Subscribes to all sensor and nav channels. Runs sensor updates → planning pipeline. Publishes OA_COMMAND at 5 Hz and OA_GRIDMAP at ~1 Hz. Source: src/acfr/oa-mapper/, config key acfr.oa-mapper.

Downstream consumers:

  • local-planner (local-planner-seeker-sitl) — receives OA_COMMAND; enabled via use_oa_mapper = true; passes vertical control to thruster-controller (minbus-control, vx/vz PID)
  • oa-map-visualizer.py — receives OA_GRIDMAP; WebSocket + HTTP; http://localhost:8050
The OA mapper integrates with the existing vehicle stack through a single config flag: use_oa_mapper = true in local-planner-seeker-sitl. When false, the mapper still runs and publishes but the planner ignores OA_COMMAND and uses its legacy altitude logic.

Occupancy Grid

The grid is a 2-D probabilistic map in the vehicle's along-heading × depth frame. Cells store occupancy probability in [0, 1]. The prior is 0.5 (unknown). A cell is considered occupied when its probability exceeds occ_thresh (default 0.62).

Motion tracking

The grid scrolls with the vehicle. On each nav update:

  • advance(ds) — shift columns left by ds m progress; new columns on the right initialise to prior 0.5
  • shift_depth(z) — re-centre depth axis on vehicle depth; new rows initialise to prior 0.5
  • A teleport guard resets the grid if the jump exceeds the grid length

Stale-voxel clearing

Each occupied cell records the vehicle heading at the time it was observed (voxel_heading_). On every update tick, cells whose recorded heading differs from the current heading by more than stale_heading_threshold_deg (default 45°) are reset to the prior. This ensures that on a heading reversal (e.g. lawnmower turn), terrain seen from the opposite direction does not persist and generate ghost obstacles.

Cells directly beneath the vehicle body (within ±vehicle_length/2 in the along-track direction from cx) are explicitly not cleared — their heading stamp is refreshed to the current heading on every tick. This preserves bottom terrain observations under the vehicle across turns, preventing the AUV from losing its altitude reference immediately after changing heading.

Sensor Models

All three sensors use the same probabilistic ray-marching approach: free cells along the ray path receive a miss decrement; the endpoint receives a hit increment or miss decrement depending on whether a surface was detected.

DVL — NUCLEUS.BOTTOMTRACK

3 beams at 20° slant angle, evenly spaced at 0° / 120° / 240° heading offsets. Only the forward-axis beam (0° offset) is axis-aligned; it alone clears voxels along its path. Lateral beams only mark their endpoint to avoid clearing voxels the 3-D beam never passed through in the projected 2-D plane. Also provides a direct altitude estimate from the minimum vertical component of surface-hitting beams.

Altimeter — NUCLEUS.ALTIMETER

Straight-down beam at vehicle X position. Ray-marches vertically through the depth column. Hit when range < max_range − 0.05 m. Miss (full range) clears along the path and marks the endpoint as free. The altimeter altitude estimate is combined with the DVL estimate in ObstacleMapper::get_altitude() — the minimum of both valid readings is reported.

Fwd Sonar — ISA500_FWD

Fan of 7 rays spanning ±3° vertical half-angle (6° total cone), fired from the vehicle nose forward in the heading direction. Surface reflection filter: measurements are silently dropped when vehicle depth < sonar_min_depth_m (1 m) to reject near-surface returns in shallow water. Updates grid cells at the nose position (not vehicle body centre), matching the physical sensor location.

Planning Pipeline

OccupancyMap::update() is called on every sensor callback and on every pose update. It runs four steps in sequence and returns the commanded depth at the vehicle column.

Step 1 — clear_stale_voxels(heading)

Reset cells observed from headings differing >45° from current heading.

Each occupied cell stores the vehicle heading at the time it was observed (voxel_heading_). Cells observed from a heading that now differs by more than stale_heading_threshold_deg (45°) are reset to the prior 0.5. Cells under the vehicle body (±vehicle_length/2 in x) are refreshed to the current heading first, so they are never stale even when turning.

Step 2 — build_cliff_manifold()

Extract the shallowest occupied voxel per column → manifold_z[].

Pass 1: find the shallowest occupied cell (p > occ_thresh) per column.

Pass 2 (behind vehicle, columns 0…cx−1): use the observation if present; otherwise default to the grid bottom.

Pass 3 (at and ahead of vehicle, columns cx…nx−1): use the observation if present; otherwise forward-extend the last seen manifold depth. This means the planner can already predict the needed altitude change before the vehicle reaches an observed cliff. Columns with no observation at all default to grid bottom.

Step 3 — build_commanded_depth(vehicle_depth)

Compute target depth per column: baseline altitude + obstacle latch + safety cap.

Step 3a — baseline: for each column, cmd_depth = max(0, manifold_z − imaging_altitude). At the vehicle column, the DVL altitude estimate is also used to adjust cmd_depth upward (but never shallower than the manifold-based value).

Step 3b — cliff latch: if the forward-obstacle scan finds a rise within the cliff_standoff window, a latch commits to a single target depth equal to the peak minus imaging_altitude. The latch overrides all forward columns until the vehicle physically clears the obstacle.

Step 3c — safety tail cap: if terrain is in the safety_below_m band behind the vehicle, cap all forward cmd_depth values at the current vehicle depth (prevents descent into terrain behind the tail).

Step 3d — propagate: unobserved forward columns inherit the vehicle-column depth so the planner always has a sensible target even without forward sonar data.

Step 4 — build_path_waypoints()

Convert cmd_depth[] into a piecewise-linear waypoint list. Currently used for visualization only.

Walks cmd_depth from cx to nx. When the depth changes by more than 2×dz between adjacent columns, a horizontal segment is emitted at the previous depth (to allow lateral clearance at the cliff edge) before a vertical segment at the new depth, producing a staircase-shaped (world_x, depth) polyline.

Note: the waypoints are computed and stored in path_waypoints_ but are not currently published in OA_COMMAND or OA_GRIDMAP and are not consumed by the local planner. The planner acts solely on the scalar vertical_mode / vertical_target / vx fields in OA_COMMAND. The waypoint list is retained as a foundation for a future path-following interface.

Cliff-top latch

The latch is the core mechanism for handling forward cliffs. When the forward-obstacle scan detects a rise in the look-ahead window it commits to a target depth equal to the shallowest detected obstacle minus imaging_altitude. The latch persists until the vehicle has physically passed the obstacle peak plus a cliff_standoff + vehicle-length margin. This prevents premature altitude resumption before the tail has cleared the obstacle.

An additional heading guard releases the latch early if the vehicle has turned >45° from the heading at latch commit, which indicates the occupancy voxels disappeared due to a heading change (lawnmower turn), not because the obstacle was cleared.

Safety tail cap

Before finalising commanded depth, a separate scan checks the terrain band immediately behind the vehicle within safety_standoff_m. If terrain in this region falls within safety_below_m of the current vehicle depth, the commanded depth for all forward columns is capped at the current vehicle depth — preventing the planner from issuing a descent command that would drive the tail into terrain.

Control Modes

The mode is updated at the end of every planning cycle based on the current latch state and depth error.

State machine transitions

[Diagram: five states — ALT_FOLLOW (top centre), OBSTACLE_HOLD (bottom left), OBSTACLE_CLEAR (bottom centre), ALT_CORRECTION (bottom right), TAIL_CLEAR (top right). All non-ALT_FOLLOW states have return arrows to ALT_FOLLOW at bottom.]

Transitions out of ALT_FOLLOW:

  • → OBSTACLE_HOLD: cliff latch active, |dz| < 1 m
  • → OBSTACLE_CLEAR: cliff latch active, dz > 1 m shallower
  • → ALT_CORRECTION: cmd_depth > 1 m deeper
  • → TAIL_CLEAR: tail safety band blocked

Transitions out of OBSTACLE_HOLD:

  • → OBSTACLE_CLEAR: latch target > 1 m shallower
  • → ALT_FOLLOW: latch released (distance cleared OR heading change >45°)

Transitions out of OBSTACLE_CLEAR:

  • → OBSTACLE_HOLD: depth error < 1 m (latch still active, vehicle has ascended to target)
  • → ALT_FOLLOW: latch released

Transitions out of ALT_CORRECTION:

  • → ALT_FOLLOW: error ≤ hysteresis

Transitions out of TAIL_CLEAR:

  • → ALT_FOLLOW: tail band clear

Mode descriptions

ALT_FOLLOW

Default mode. No latch active, depth error within altitude_overshoot_threshold_m (1 m). Vehicle follows terrain at imaging altitude.

  • vertical_mode: ALT_FOLLOW
  • vertical_target: imaging_altitude (2 m)
  • vx: survey_speed (0.5 m/s)

OBSTACLE_HOLD

Cliff latch active; depth error < altitude_overshoot_threshold_m (1 m). Vehicle holds depth at the latched target while advancing forward at survey speed. The vehicle does not ascend in this mode — OBSTACLE_CLEAR is selected instead when the target is more than 1 m shallower than the vehicle.

  • vertical_mode: DEPTH_HOLD
  • vertical_target: cmd_depth at vehicle column (latched target)
  • vx: survey_speed

OBSTACLE_CLEAR

Cliff latch active; latched target > 1 m shallower than vehicle. Vehicle must ascend to clear the obstacle. Forward motion is paused (vx = 0) until within 0.1 m of the target depth, then survey_speed resumes. The typical path is OBSTACLE_CLEAR → OBSTACLE_HOLD (as vehicle ascends to target, error drops below 1 m) → ALT_FOLLOW (when latch releases).

  • vertical_mode: DEPTH_HOLD
  • vertical_target: cmd_depth at vehicle column (latched target)
  • vx: 0 m/s until within 0.1 m of target, then survey_speed

ALT_CORRECTION

No latch active; cmd_depth > altitude_overshoot_threshold_m (1 m) deeper than vehicle (altitude is too high, vehicle needs to descend). Forward motion is paused until depth is corrected.

  • vertical_mode: DEPTH_HOLD
  • vertical_target: cmd_depth at vehicle column
  • vx: 0 m/s until depth reached

TAIL_CLEAR

No latch active; obstacle in tail safety band (behind vehicle body, within safety_below_m depth of vehicle). Vehicle holds current depth and continues at survey speed to move forward away from the terrain behind its tail.

  • vertical_mode: DEPTH_HOLD
  • vertical_target: current vehicle depth
  • vx: survey_speed

The oa-mapper binary is the only process with LCM dependencies. The core library (occupancy_map.*) is framework-free and directly usable as a C++ library or Python extension.

Testing

1. Standalone logic simulator

Before vehicle integration the algorithm was validated in a standalone Python simulator that drives the same occupancy-map core against synthetic terrain profiles — sawtooth slopes, cliffs, random reef bathymetry — at configurable sensor rates. This confirms mode transitions and altitude-following behaviour independently of any hardware or middleware.

Sawtooth terrain

Watch video: Sawtooth terrain

Repeating cliff faces — exercises the full OBSTACLE_HOLD → OBSTACLE_CLEAR → OBSTACLE_HOLD → ALT_FOLLOW cycle and verifies the latch commits and releases correctly on each peak.

Continuous slope

Watch video: Continuous slope

Gradually rising and falling terrain — confirms smooth altitude following (ALT_FOLLOW) and correct triggering of ALT_CORRECTION on descent without false obstacle latches.

Random terrain

Watch video: Random terrain

Stochastic bathymetry with irregular peaks and troughs — stress-tests mode stability and stale-voxel clearing under varied heading changes and sensor update rates.

2. Vehicle SITL in Gazebo

End-to-end integration test with the Seeker vehicle model in Gazebo. All three sensors (DVL, altimeter, forward sonar) feed live LCM messages into oa-mapper running inside the vehicle software container. The OA map visualizer at http://localhost:8050 provides live 2-D top-down, depth profile, and 3-D terrain views during the mission.

Watch video: AUV SITL in Gazebo with OA mapper active

Demonstrates the vehicle navigating terrain with obstacle avoidance active, showing the OA map visualizer alongside the Gazebo / RViz views.

3. Real vehicle LCM log replay

The mapper and visualizer can be driven from a recorded vehicle LCM log using lcm-logplayer, with no hardware or simulator required. This validates that the algorithm responds correctly to real sensor noise, dropout patterns, and navigation timing from an actual dive.

Replay command: lcm-logplayer --speed 1.0 <logfile.lcm>. The mapper node and visualizer start normally; the log republishes all LCM channels at recorded timing. Set speed < 1.0 to slow playback for detailed inspection.
Note: The log used here was recorded from a real dive where the vehicle was running the old altitude-following logic, not oa-mapper. The oa-mapper node was run offline against the log to demonstrate that the occupancy grid builds correctly and sensor data is processed as expected from real hardware — the OA_COMMAND output was not controlling the vehicle during this dive.
