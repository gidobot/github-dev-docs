Overview

The OA Mapper is a new probabilistic obstacle-avoidance system added to acfr-lcm for the ACFR Seeker AUV. It maintains a 2-D voxel occupancy grid anchored to the vehicle in the along-heading × depth plane, fuses observations from three onboard sensors, and produces a continuous vertical depth command that keeps the vehicle at a constant altitude above terrain while safely crossing cliffs and avoiding obstacles ahead.

Core library

auv_oavoid::OccupancyMap and ObstacleMapper in src/acfr/oa-mapper/occupancy_map.* — LCM-free C++ library containing all grid logic, sensor updates, and planning. No framework dependency.

LCM node

oa_mapper.cpp wraps the core library: subscribes to sensor and nav LCM channels, calls the planning pipeline on every update, and publishes OA_COMMAND + OA_GRIDMAP at the configured rate.

Web visualizer

src/acfr/python/oa-map-visualizer.py subscribes to OA_GRIDMAP and serves a live 3-panel view (top-down grid, depth profile, 3-D terrain) at http://localhost:8050.

Architecture

The OA mapper runs as a standalone process alongside the existing vehicle software stack. It is purely reactive — consuming sensor and nav LCM messages and publishing commands — with no shared memory or direct coupling to the planner.

The OA mapper integrates with the existing vehicle stack through a single config flag: use_oa_mapper = true in local-planner-seeker-sitl. When false, the mapper still runs and publishes but the planner ignores OA_COMMAND and uses its legacy altitude logic.

Occupancy Grid

The grid is a 2-D probabilistic map in the vehicle's along-heading × depth frame. Cells store occupancy probability in [0, 1]. The prior is 0.5 (unknown). A cell is considered occupied when its probability exceeds the occ_thresh (default 0.62).

Motion tracking

The grid scrolls with the vehicle. On each nav update:

  • advance(ds) — shift columns left by ds m progress; new columns on the right initialise to prior 0.5
  • shift_depth(z) — re-centre depth axis on vehicle depth; new rows initialise to prior 0.5
  • A teleport guard resets the grid if the jump exceeds the grid length

Stale-voxel clearing

Each occupied cell records the vehicle heading at the time it was observed (voxel_heading_). On every update tick, cells whose recorded heading differs from the current heading by more than stale_heading_threshold_deg (default 45°) are reset to the prior. This ensures that on a heading reversal (e.g. lawnmower turn), terrain seen from the opposite direction does not persist and generate ghost obstacles.

Cells directly beneath the vehicle body (within ±vehicle_length/2 in the along-track direction from cx) are explicitly not cleared — their heading stamp is refreshed to the current heading on every tick. This preserves bottom terrain observations under the vehicle across turns, preventing the AUV from losing its altitude reference immediately after changing heading.

Sensor Models

All three sensors use the same probabilistic ray-marching approach: free cells along the ray path receive a miss decrement; the endpoint receives a hit increment or miss decrement depending on whether a surface was detected.

DVL — NUCLEUS.BOTTOMTRACK

3 beams at 20° slant angle, evenly spaced at 0° / 120° / 240° heading offsets. Only the forward-axis beam (0° offset) is axis-aligned; it alone clears voxels along its path. Lateral beams only mark their endpoint to avoid clearing voxels the 3-D beam never passed through in the projected 2-D plane.

Also provides a direct altitude estimate from the minimum vertical component of surface-hitting beams.

Altimeter — NUCLEUS.ALTIMETER

Straight-down beam at vehicle X position. Ray-marches vertically through the depth column. Hit when range < max_range − 0.05 m. Miss (full range) clears along the path and marks the endpoint as free.

The altimeter altitude estimate is combined with the DVL estimate in ObstacleMapper::get_altitude() — the minimum of both valid readings is reported.

Fwd Sonar — ISA500_FWD

Fan of 7 rays spanning ±3° vertical half-angle (6° total cone), fired from the vehicle nose forward in the heading direction. Surface reflection filter: measurements are silently dropped when vehicle depth < sonar_min_depth_m (1 m) to reject near-surface returns in shallow water.

Updates grid cells at the nose position (not vehicle body centre), matching the physical sensor location.

The sonar uses lower probabilities because it is noisier (ISA500 single-beam range sonar vs. Nucleus acoustic Doppler). A hit still drives cells well above the 0.62 threshold after ~2 consecutive detections.

Planning Pipeline

OccupancyMap::update() is called on every sensor callback and on every pose update. It runs four steps in sequence and returns the commanded depth at the vehicle column.

Cliff-top latch

The latch is the core mechanism for handling forward cliffs. When the forward-obstacle scan detects a rise in the look-ahead window it commits to a target depth equal to the shallowest detected obstacle minus imaging_altitude. The latch persists until the vehicle has physically passed the obstacle peak plus a cliff_standoff + vehicle-length margin. This prevents premature altitude resumption before the tail has cleared the obstacle.

An additional heading guard releases the latch early if the vehicle has turned >45° from the heading at latch commit, which indicates the occupancy voxels disappeared due to a heading change (lawnmower turn), not because the obstacle was cleared.

Safety tail cap

Before finalising commanded depth, a separate scan checks the terrain band immediately behind the vehicle within safety_standoff_m. If terrain in this region falls within safety_below_m of the current vehicle depth, the commanded depth for all forward columns is capped at the current vehicle depth — preventing the planner from issuing a descent command that would drive the tail into terrain.

LCM Integration

The oa-mapper binary is the only process with LCM dependencies. The core library (occupancy_map.*) is framework-free and directly usable as a C++ library or Python extension.

Testing

1. Standalone logic simulator

Before vehicle integration the algorithm was validated in a standalone Python simulator that drives the same occupancy-map core against synthetic terrain profiles — sawtooth slopes, cliffs, random reef bathymetry — at configurable sensor rates. This confirms mode transitions and altitude-following behaviour independently of any hardware or middleware.

Sawtooth terrain
Repeating cliff faces — exercises the full OBSTACLE_HOLD → OBSTACLE_CLEAR → OBSTACLE_HOLD → ALT_FOLLOW cycle and verifies the latch commits and releases correctly on each peak.
Continuous slope
Gradually rising and falling terrain — confirms smooth altitude following (ALT_FOLLOW) and correct triggering of ALT_CORRECTION on descent without false obstacle latches.
Random terrain
Stochastic bathymetry with irregular peaks and troughs — stress-tests mode stability and stale-voxel clearing under varied heading changes and sensor update rates.

2. Vehicle SITL in Gazebo

End-to-end integration test with the Seeker vehicle model in Gazebo. All three sensors (DVL, altimeter, forward sonar) feed live LCM messages into oa-mapper running inside the vehicle software container. The OA map visualizer at http://localhost:8050 provides live 2-D top-down, depth profile, and 3-D terrain views during the mission.

3. Real vehicle LCM log replay

The mapper and visualizer can be driven from a recorded vehicle LCM log using lcm-logplayer, with no hardware or simulator required. This validates that the algorithm responds correctly to real sensor noise, dropout patterns, and navigation timing from an actual dive.

Replay command: lcm-logplayer --speed 1.0 <logfile.lcm>
The mapper node and visualizer start normally; the log republishes all LCM channels at recorded timing. Set speed < 1.0 to slow playback for detailed inspection.
The log used here was recorded from a real dive where the vehicle was running the old altitude-following logic, not oa-mapper. The oa-mapper node was run offline against the log to demonstrate that the occupancy grid builds correctly and sensor data is processed as expected from real hardware — the OA_COMMAND output was not controlling the vehicle during this dive.
